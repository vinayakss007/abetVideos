# Lazy-loaded video editor with server-side processing

A full post-production editor that loads on demand and delegates all heavy work to the backend. The frontend is a React.lazy chunk (EditorPage + six sub-components + useEditor hook) that mounts at `/edit/:videoId`, fetches scene metadata saved during video assembly, and sends edit instructions (reorder, trim, overlay, audio levels) to a new set of FastAPI endpoints which apply them via MoviePy. Frame preview is extracted server-side and returned as base64 JPEG. The editor unloads cleanly when the user navigates away.

**Watch for:** The `apply_edits` endpoint holds the full source video in RAM and re-encodes synchronously on the request thread -- a 10-minute video will block the worker for minutes and likely OOM under concurrent use (confirmed). The `SceneTrim` schema allows `end_time < start_time`, which the backend silently produces a zero-length clip for but the frontend shows negative duration (confirmed). Media swap only updates the frontend scene list -- the swap is never transmitted to the backend as part of `EditInstruction`, so it's discarded on submit (confirmed).

## High-level view

The code-splitting strategy works as intended: Vite produces a separate `EditorPage-*.js` chunk, loaded only when the route is visited. The `useEditor` hook's `mountedRef` prevents post-unmount state updates, and the cleanup in `EditorPage`'s effect calls `editor.reset()`. This meets the RAM-efficiency requirement for the frontend.

The backend introduces three endpoints behind a regex-validated video ID path parameter (`^[a-f0-9]{12}$`). `GET /scenes` reads a `_meta.json` file produced by the assembler; `POST /edit` loads the source video, splits by stored scene boundaries, applies transforms, re-encodes to a new file; `POST /preview-frame` extracts a single frame. All MoviePy work runs synchronously in the async endpoint handlers, meaning the entire event loop blocks during encoding.

The `video_editor.py` service is careful about clip lifecycle (try/finally closing all clips), but it processes the full video in-memory and writes with `threads=2`. There's no concurrency guard, timeout, or background-task pattern, so two simultaneous edit requests on a 10-minute source could exhaust RAM.

The frontend editor UI is DOM-only as required -- the text overlay preview uses absolutely-positioned `<span>` elements, the preview panel renders `<img>` tags with base64 src, and the timeline uses native HTML5 drag-and-drop. No canvas or WebGL dependencies exist.

<details>
<summary>Issues (6)</summary>

1. **Edit endpoint blocks the event loop** -- `apply_edits` runs MoviePy encoding synchronously inside an `async def` handler. Wrap it in `asyncio.to_thread` (as the assembler already does) to avoid blocking all other requests during a render.
2. **No concurrency or memory guard on editing** -- Two concurrent edit requests on large videos will double RAM usage with no backpressure. Add a semaphore or queue, or at minimum document the constraint.
3. **SceneTrim allows end < start** -- The Pydantic model validates both fields are `>= 0` but never asserts `end_time > start_time`. The backend clamps silently, but the frontend displays negative durations in the trimmer.
4. **Media swap is client-only** -- `MediaSwapPanel.onSwap` updates the scene array in React state, but `EditInstruction` has no field to convey the new media URL to the server. The swap is lost on submit.
5. **Preview timestamp exceeds scene boundary** -- `PreviewPanel` constrains the slider to `scene.duration_seconds`, but sends an absolute timestamp to the backend. If the scene starts at 30s into the video, requesting timestamp=5 shows a frame from scene 1, not the selected scene.
6. **Editor state leak between videos** -- If the user navigates from `/edit/aaa` to `/edit/bbb` without an intervening unmount (React Router reuses the component), the `useEffect` cleanup runs but `editor.reset()` races with the new `loadScenes` call because both fire in the same render cycle.

## Synchronous encoding in async handlers

The `edit_video` endpoint in `editor.py` calls `apply_edits(...)` directly:

```python
async def edit_video(request: EditRequest, video_id: str = Path(...)):
    ...
    result = apply_edits(video_path=str(video_path), ...)
```

`apply_edits` calls `final.write_videofile(...)` which is a CPU-bound FFmpeg subprocess that runs for seconds to minutes. During that time, the FastAPI event loop is completely blocked -- no other request can be served. The assembler already solved this with `await asyncio.to_thread(final.write_videofile, ...)`. The same pattern should be applied here. (confirmed -- line 77 of `editor.py` vs line 319 of `video_assembler.py`)

The same issue applies to `extract_frame`, which opens a `VideoFileClip` and calls `get_frame()`. Less severe (sub-second for most videos) but still blocking.

## SceneTrim schema gap

```python
class SceneTrim(BaseModel):
    scene_number: int
    start_time: float = Field(..., ge=0)
    end_time: float = Field(..., ge=0)
```

Nothing prevents `start_time=5.0, end_time=2.0`. The backend code handles it gracefully (`scene_end = max(scene_start, min(scene_end, end))` produces a zero-length clip that's skipped), but the frontend `SceneTrimmer` displays `(end - start).toFixed(1)` which shows "-3.0s". Add a `model_validator` enforcing `end_time >= start_time`, or at minimum a frontend clamp.

## Media swap has no backend contract

`MediaSwapPanel` calls `onSwap(item.url, item.media_type)` which updates the scene's `media_url` in the React state. But `EditInstruction` only carries `scene_order`, `trims`, `text_overlays`, `audio_levels`, and `background_music_volume`. There's no field for media replacement. When the user clicks "Save Edits", the swap instruction is silently dropped. Either add a `media_replacements` field to `EditInstruction` and handle it in `apply_edits`, or remove the panel until the backend supports it.

Additionally, the `onSwap` callback signature in `EditorPage.tsx` only accepts `(url: string)` while `MediaSwapPanel` passes `(item.url, item.media_type)` -- the second argument is silently ignored due to JavaScript's arity tolerance, but the TypeScript interface mismatch means the `media_type` string is never captured.

## Preview timestamp is video-global, not scene-relative

`PreviewPanel` uses the slider value as the `timestamp` POST body to `/preview-frame`. This is the video-global timestamp, not scene-relative. The slider range is `[0, scene.duration_seconds]`, so for a 5-second scene that starts at 30s in the source video, scrubbing to 3s sends `timestamp: 3.0` and the backend returns the frame at second 3 of the video (which belongs to scene 1, not the selected scene). To fix: compute the absolute offset by accumulating durations of preceding scenes, or add a `scene_number` param and let the backend resolve it.

## Editor state when route changes without unmount

React Router reuses the `EditorPage` component instance when the path parameter changes (e.g. navigating from one video's editor to another). The `useEffect` dependency is `[videoId]`, so when `videoId` changes:

1. The cleanup of the previous effect runs (`cancelled = true; editor.reset()`)
2. The new effect fires immediately (`loadScenes()` for the new videoId)

Because `editor.reset()` is a state setter (enqueues a re-render) and `loadScenes()` also sets state, both execute in the same commit. If `reset()` dispatches after the new scenes are loaded, it wipes them. The `cancelled` flag only guards the previous `loadScenes` promise -- it doesn't protect against `reset()` being batched after the new data arrives. In practice this is unlikely (the fetch takes network time), but under fast cache responses it's a real race.

## Test coverage

Backend tests cover schema validation thoroughly (all boundary cases for ranges, negative values, out-of-bound fields) and endpoint routing (404/422 responses, mocked success paths). Frontend tests cover lazy loading, error states, and SceneTimeline rendering.

Not tested: the actual `apply_edits` service function with real MoviePy operations (all endpoint tests mock the service layer). The `_apply_text_overlay` function, trim clamping logic, and scene boundary splitting are untested at the unit level. The `video_assembler.py` metadata-saving path added in this diff is also untested.
