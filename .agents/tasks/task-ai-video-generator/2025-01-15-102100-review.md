# AI video generation pipeline (v2)

A full-stack application that generates narrated videos from a topic: LLM script generation, edge-tts voice-over, concurrent stock media sourcing (Pexels/Pixabay/Giphy), and MoviePy video assembly. FastAPI backend with step-by-step REST endpoints and a streaming SSE full-pipeline endpoint. React + Vite + TypeScript wizard frontend with Tailwind dark theme. Docker Compose with nginx reverse proxy.

Since v1: CORS credentials removed, frontend API client payload shapes fixed, TypeScript types aligned with backend schemas, style enum values corrected to match backend, temp file cleanup added to video assembler, media sourcing made concurrent with `asyncio.gather`, SSE errors now include `failed_stage`.

**Watch for:** Path traversal in the video download endpoint allows reading arbitrary `.mp4` files outside the output directory (confirmed). The `write_videofile` call in the assembler is synchronous and blocks the async event loop for the entire encoding duration, starving all other requests (confirmed). Concurrent media sourcing can fire dozens of simultaneous HTTP requests with no semaphore, risking API rate-limit bans on Pexels/Pixabay (likely).

## High-level view

The video download endpoint accepts any string as `video_id` and passes it directly into a path join, allowing `../../` traversal to read any `.mp4`-suffixed file the process can access. This is the highest-severity item.

The video assembler's `write_videofile` is a CPU-intensive synchronous call that runs inline in the async handler. For a multi-minute video, this blocks the event loop for tens of seconds, during which no other SSE events, health checks, or API responses can be served. The fix is to run it in a thread pool via `asyncio.to_thread` or `loop.run_in_executor`.

Media sourcing now uses `asyncio.gather` with a shared client, which is a good improvement over sequential fetching. However, for a 10-minute video with 30 scenes, this fires up to 30 parallel search-then-download chains against the same APIs with no concurrency limit, increasing the chance of hitting rate limits.

The frontend-backend contract is now aligned for the step-by-step wizard path. Types, payload shapes, and enum values all match.

Test coverage remains focused on the routing/serialization layer via mocked service functions. The actual media download logic, TTS subprocess management, and MoviePy composition paths have no automated tests.

<summary>Issues (5)</summary>

1. **Path traversal in video download** — `video_id` is user-supplied and joined directly to `output_dir / "videos"` without sanitization. An attacker can request `../../etc/passwd.mp4` or similar. Validate that `video_id` matches `^[a-f0-9]{12}$` or resolve the path and confirm it's under the videos directory.
2. **Blocking event loop during video encoding** — `final.write_videofile(...)` is synchronous, blocking the entire async event loop for the duration of encoding. Wrap it in `asyncio.to_thread()` or run the assembler in a subprocess.
3. **Unbounded concurrent media requests** — `asyncio.gather` on all scenes fires an unlimited number of parallel HTTP requests. Add a semaphore (e.g., `asyncio.Semaphore(5)`) to cap concurrency and avoid API rate-limit bans.
4. **MoviePy clips not closed on assembly error** — If `write_videofile` or concatenation raises, the `scene_clips` list and their audio sub-clips are never closed, leaking file descriptors and temp files until GC collects them.
5. **No integration test for error-stage propagation** — The SSE error test verifies the message text but not the `failed_stage` field in the error data payload. Add an assertion for `error_events[0]["data"]["failed_stage"]`.

<summary>Details</summary>

## Path traversal in video download

The download endpoint constructs a file path by direct concatenation:

```python
video_path = output_dir / f"{video_id}.mp4"
```

`video_id` arrives from the URL path parameter with no validation beyond what FastAPI's `str` type provides. A request to `/api/videos/..%2F..%2F..%2Fetc%2Fpasswd.mp4/download` (URL-decoded by the framework) would resolve outside the intended directory. Even though it appends `.mp4`, the path traversal still allows reading any `.mp4`-suffixed file on the filesystem.

The fix is either a regex constraint on the path parameter (`video_id: str = Path(pattern=r"^[a-f0-9]{12}$")`) or resolving the path and verifying `.resolve().parent == output_dir / "videos"`.

## Synchronous video encoding blocks the event loop

`assemble_video` is declared `async` but performs all MoviePy operations synchronously on the calling coroutine. The critical bottleneck is `final.write_videofile(...)`, which for a 5-minute video can take 30-60 seconds of pure CPU work. During this time, the single event loop thread is blocked:

- The SSE endpoint cannot yield progress events to other clients.
- Health check responses are delayed.
- Any concurrent REST requests queue behind the blocked loop.

Since `assemble_video` is called from both the SSE pipeline and the `/assemble` REST endpoint, this affects all consumption paths. The function should run in a thread executor or use MoviePy's subprocess-based writing to avoid loop starvation.

## Unbounded concurrency in media sourcing

The `source_media` function gathers all scenes at once:

```python
results = await asyncio.gather(*tasks)
```

For a 10-minute video the LLM might generate 20-30 scenes. Each scene fires up to 4 sequential API searches (Pexels video, Pixabay video, Pexels photo, Pixabay image) with fallback logic plus a download. With 30 scenes in parallel, this means potentially 120 outbound API calls within seconds. Both Pexels (200/hour) and Pixabay rate limits could be hit for longer videos. A semaphore limiting to 3-5 concurrent scene-sourcing tasks would prevent this without significantly increasing total latency.

## Resource cleanup on assembly failure

The assembler closes clips in the happy path but has no `try/finally` around the `write_videofile` call:

```python
final.write_videofile(...)
total_duration = final.duration
final.close()
for clip in scene_clips:
    clip.close()
```

If `write_videofile` or `concatenate_videoclips` raises (e.g., codec not found, disk full, corrupt media file), all opened `AudioFileClip`, `VideoFileClip`, and `ImageClip` objects remain unclosed. Each holds an open file descriptor and potentially a subprocess (for video decoding). Under repeated failures this leaks resources until the process is restarted. Wrapping the assembly in `try/finally` that closes all clips would prevent this.

## Test coverage gaps remain

The integration tests mock all service functions at the router boundary. This verifies serialization, routing, and HTTP status codes but leaves the service internals untested:

- `media_sourcer`: no test for the download flow, keyword extraction, or fallback logic when APIs return empty results.
- `tts_service`: no test for ffprobe duration parsing, edge-tts invocation, or behavior when ffprobe is missing.
- `video_assembler`: no test for the clip-building functions, loop behavior, or cleanup logic.

The SSE error-handling test checks the error message but doesn't assert on the `failed_stage` field that was added in this revision. Adding `assert error_events[0]["data"]["failed_stage"] == "script_generation"` would verify the new feature.
