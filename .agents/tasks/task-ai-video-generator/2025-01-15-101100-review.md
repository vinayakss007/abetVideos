# Full-stack AI video generation pipeline

A full-stack application that takes a topic, generates a narrated video script via OpenAI-compatible LLM, converts narration to audio (edge-tts), sources stock footage/images/GIFs from Pexels/Pixabay/Giphy, and assembles the final MP4 with MoviePy. The backend is FastAPI with both step-by-step REST endpoints and a single SSE-streaming full-pipeline endpoint. The frontend is a React + Vite + TypeScript wizard UI with Tailwind dark theme. Docker Compose ties the two services together behind nginx.

**Watch for:** Frontend-backend schema mismatch will cause the step-by-step flow to fail at runtime (confirmed). The `assembleVideo` call in the client sends fields the backend doesn't expect, and the `generateTTS` call wraps the script in a different shape than the endpoint requires. Frontend style enum values (`entertainment`, `tutorial`) don't match backend enum values (`entertaining`, `documentary`, `motivational`), causing 422 errors for 2 of 4 style options (confirmed). No cleanup of temporary media/audio files ever runs, so disk usage grows unboundedly (confirmed). CORS is `allow_origins=["*"]` with `allow_credentials=True`, which is invalid per the Fetch spec (confirmed).

## High-level view

The pipeline has two consumption paths: a step-by-step wizard (individual REST endpoints) and a single SSE streaming endpoint (`/generate-full`). The SSE path works because it runs server-side without frontend serialization concerns, but the step-by-step wizard path is broken by payload shape mismatches between the frontend API client and backend Pydantic models.

The frontend offers style and duration options that partially diverge from the backend's enum definitions. Two of four style choices will produce validation errors on submission.

Media sourcing runs sequentially per scene with a fresh HTTP client per scene, making it the latency bottleneck for longer videos. All intermediate files (media downloads, TTS audio) persist indefinitely with no cleanup.

Security posture is development-grade: wildcard CORS with credentials, no auth on any endpoint, no rate limiting. Anyone who can reach the backend can burn through API quota on OpenAI, Pexels, Pixabay, and Giphy.

<details>
<summary>Issues (8)</summary>

1. **Frontend assembleVideo payload mismatch** — The client sends `{ script, audio_files, media_items }` but the backend's `AssembleVideoRequest` expects `{ script, tts_results, scene_media, format }`. This will produce a 422 on every step-by-step assembly call. Fix the client payload shape.
2. **Frontend generateTTS payload mismatch** — The client sends the script object directly (`POST /videos/generate-tts`, body = script) but the backend expects `{ script: ..., voice?: ... }` (a `GenerateTTSRequest` wrapper). Will produce a 422. Wrap the script in `{ script: ... }`.
3. **Frontend type divergence from backend** — `MediaItem.type` on the frontend vs `MediaItem.media_type` on the backend; `TTSResult.duration` vs `TTSResult.duration_seconds`; `VideoResult.duration` vs `VideoResult.duration_seconds`. These mismatches mean step-by-step responses won't map correctly to UI state.
4. **Frontend style enum mismatch** — TopicInput offers `entertainment` and `tutorial`; backend accepts `entertaining`, `documentary`, `motivational`. Two of four style buttons will fail validation. Align the frontend options with the backend enum.
5. **No temp file cleanup** — Downloaded media, TTS audio, and assembled videos accumulate in `./output` with no expiry or cleanup. Add a periodic cleanup or on-completion cleanup for intermediate files.
6. **CORS wildcard with credentials** — `allow_origins=["*"]` combined with `allow_credentials=True` is invalid per the CORS spec (browsers will reject credentialed requests with `*` origin). Either drop credentials or enumerate allowed origins.
7. **Pixabay API key in query string** — The Pixabay key is sent as a `key=` query parameter, which means it appears in server access logs, proxy logs, and CDN caches. This is a Pixabay API requirement, but document the risk.
8. **Sequential media sourcing** — Each scene is sourced sequentially with its own `httpx.AsyncClient` instance. For longer videos (10+ scenes), this multiplies latency. Use `asyncio.gather` with a shared client.

## Frontend-backend contract divergence

The API client (`frontend/src/api/client.ts`) builds requests that don't match the backend Pydantic models:

```typescript
// client.ts line 35 — sends the raw script object as body
export async function generateTTS(script: VideoScript): Promise<TTSResult[]> {
  const response = await apiClient.post<TTSResult[]>('/videos/generate-tts', script);
  return response.data;
}
```

The backend expects `GenerateTTSRequest` which wraps the script: `{ "script": { ... }, "voice": null }`. Sending the bare script means pydantic will try to find a `script` field inside the script's own properties and fail.

Similarly, `assembleVideo` sends `{ script, audio_files, media_items }` but the backend expects `{ script, tts_results, scene_media, format }`. The field names don't align at all.

The TypeScript types also diverge from backend schemas: `MediaItem` uses `type` where the backend uses `media_type`; `TTSResult` has `duration` where the backend has `duration_seconds`; `VideoResult` has `duration` where the backend has `duration_seconds`. The SSE full-pipeline path bypasses the client functions and casts `event.data` directly, so it may appear to work, but the step-by-step wizard flow is broken end-to-end.

## Frontend style enum doesn't match backend

`TopicInput.tsx` offers: `educational`, `entertainment`, `news`, `tutorial`. The backend's `VideoStyle` enum accepts: `educational`, `entertaining`, `documentary`, `motivational`, `news`. Only `educational` and `news` overlap. Submitting `entertainment` or `tutorial` will produce a 422 validation error from the `/generate-script` endpoint.

## Disk accumulation and resource lifecycle

Every pipeline run creates:
- `output/media/<uuid>/scene_NNN/` with downloaded media files
- `output/tts/<uuid>/scene_NNN.mp3` with audio files
- `output/videos/<video_id>.mp4` as the final output

None of these intermediate directories are cleaned up after video assembly completes. The `video_assembler.py` closes MoviePy clips but doesn't remove the source media or audio files. For 10-minute videos with 30+ scenes, each run could accumulate hundreds of MB of intermediate files.

The `source_media_for_scene` function creates a new `httpx.AsyncClient()` per scene rather than sharing one across scenes, adding connection setup overhead. Combined with the sequential loop in `source_media`, media sourcing latency scales linearly with scene count.

## SSE error reporting is stage-blind

The SSE endpoint catches all exceptions in a single `try/except` and emits one error event with `"step": "error"`. If the failure happens mid-pipeline (e.g., TTS succeeds but media sourcing fails), the client receives only `"Pipeline failed: <message>"` with no indication of which stage failed or what partial results were produced. For debugging and UX, the error event should include the stage that was running when the failure occurred.

## CORS and open access

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    ...
)
```

Per the Fetch spec, `Access-Control-Allow-Origin: *` with `Access-Control-Allow-Credentials: true` is invalid. Browsers will reject credentialed cross-origin requests when the origin header is `*`. Since there's no auth currently, this is latent rather than immediately breaking, but the configuration is wrong regardless.

The backend has no auth on any endpoint and no rate limiting. Anyone who can reach port 8000 can trigger pipeline runs that consume third-party API quota. The video download endpoint accepts any 12-character hex ID with no access control.

## Video assembler edge case with negative padding

`concatenate_videoclips(..., padding=-0.5)` applies a 0.5-second cross-fade between scenes. If a TTS audio file is shorter than 0.5 seconds (possible with very short narration text), the negative padding could produce a zero-or-negative-duration overlap, which MoviePy would reject at write time. This is unlikely given the 2-second minimum `duration_seconds` in the schema, but edge-tts could produce audio shorter than the specified duration for very few words.

## Test coverage gaps

Not tested: media sourcer functions (API calls, download logic, keyword extraction), TTS service (edge-tts integration, ffprobe duration parsing), video assembler (MoviePy composition). These are the most complex and failure-prone parts of the system. The frontend tests verify component rendering but not user interaction flows or the step-by-step API integration that is broken by the payload mismatches.
